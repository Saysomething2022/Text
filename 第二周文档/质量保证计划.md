# 质量保证计划

## 1. 主题内容与适用范围

&nbsp;&nbsp;&nbsp;&nbsp;本规范规定了在制定软件质量保证计划时应该遵循的统一的基本要求。  

&nbsp;&nbsp;&nbsp;&nbsp;本规范适用于软件特别是重要软件的质量保证计划制订工作。对于非重要软件或已经开发好的软件，可以采用本规范规定的要求的子集。

## 3.质量目标
  - 达到监理和制定标准 
  - 关注过程控制，减少施工过程中的质量缺陷
  - 将无缺陷作为质量目标
  - 通过质量缺陷预防措施，及时关闭整改项，以实现质量目标

| 用户 | user | 用户是指实际使用软件来完成某项计算、控制或数据处理等任务的单位或个人。 |

| 软件质量 | software quality（SQ） | 软件质量是指软件产品中能满足给定需求的各种特性和总和。这些特性称做质量特性，它包括功能度、可靠性、时间经济性、资源经济性、可维护性和或移植性等。 |
| 质量保证 | quality assurance（QA） | 质量保证是指为使软件产品规定需求所进行的一系列有计划的必要工作。 |

## 5. 软件质量保证计划编制大纲
### 5.1. 引言

本质量保障计划的创立目的为规范程序的质量要求，规范在软件开发过程中的测试流程。详细说明在满足要求情况下的产品的测试要求和程序，以确保最终的成品与需求的样品完全一致。

### 5.2. 管理

在软件开发过程中，本组的组长负责软件代码总体质量。在成员编写后端服务时，所有服务类必须有对应的单元测试类，一并提交至远端Github，在将成员的代码合并到主分支前，代码必须通过持续集成编译和自动运行的单元测试才能发起和并请求。如果成员的代码未能通过单元测试，则需要重新提交。此外，组长负责审核和并请求并执行合并。

### 5.4. 其他文档

除基本文档以外，还应包括下列文档：

- 接口文档  
  &nbsp;&nbsp;&nbsp;&nbsp;在进行前后端对接时，必须编写完整的API接口文档，所有的业务逻辑操作流程由后端项目负责人测试，必须满足在所有允许的业务逻辑操作下都能正确的反应文档中的含义。  
  &nbsp;&nbsp;&nbsp;&nbsp;我们的所有API采用JSON格式序列化请求，在请求的值部分必须能体现出该字段的数据意义、该字段的数据约束，在返回值部分需要能体现出所有可能性的结果，如果有必要，请在最后部分注明返回值中某些数字的含义。  
  一个合理的接口文档的示例如下：
  ```
  [Get/Post] /user/login
  RequestHeader: Token
  {
      userName: "用户名字符串，长度为3-20"
      passwordSalt: "密码哈希，前端需验证密码长度为10，验证成功后使用SHA256算法计算哈希"
  }
  ---
  ResponseHeader: Status(200[1]|401[2]|402[3]), Token(If Login Success)
  {
      message: "如果登录失败，则此字段表示失败原因"
  }
  *[1]: 登录成功
  *[2]: 用户名或密码错误
  *[3]: 账号已被封禁
  ```
  - 后端JavaDoc文档  
  &nbsp;&nbsp;&nbsp;&nbsp;JavaDoc文档不需要编写一个独立的文档，需要在代码的函数前、类型前以正确的语法来解释其含义。要求至少在所有公开的工具类、Service成员函数前包含此注释，以便他人正确调用。  
  例如：
  ``` java
  /**
  * 给指定用户生成一个JWT Token
  * @param userId 用户ID
  * @param expireTimeSeconds 过期时间，秒为单位
  * @return JWT Token
  */
  public String generateJwt(int userId, int expireTimeSeconds);
  ```

### 5.8. 标准、约定、规范和度量
 
&nbsp;&nbsp;&nbsp;&nbsp;在满足质量要求的基础上制定或提出参见的质量标准。包括如下方面的标准：
1. 项目文档类型标准
《项目文档目录》中提供的各阶段需要编制的文档。
2. 评审文档的度量标准
&nbsp;&nbsp;完备性：按照《项目文档目录》中提供的各阶段规定编制相应文档，以保证在项目实施阶段结束时其文档是齐全的。
&nbsp;&nbsp;正确性：在项目实施阶段 编写的文档的内容，必须真实反映该阶段的工作且与项目的需求相一致。
&nbsp;&nbsp;简明性：在实施阶段编写的各文档的语言表达应该清晰、准确简练，适合各文档的特性读者。
&nbsp;&nbsp;可追踪性：在项目实施阶段编写的各类文档应该具有良好的可追踪性。文档的可追踪性包括纵向可追踪性与横向可追踪性两个方面。前者是指在不同文档的相关内容之间相互检索的难易程度；后者是指确定同一文档某一内容在本文档中的涉及范围的难易程度。
&nbsp;&nbsp;自说明性：在项目实施阶段编写的各文档应该具有较好的自说明性。文档的自说明性是指在项目实施阶段中的不同文档可独立表达该项目其相应阶段的阶段产品的能力。
&nbsp;&nbsp;规范性：在项目实施阶段编写的各文档应该具有良好的规范性。文档的规范性是指文档的封面、大纲、术语的含义以及图示符号等符合有关规范的规定。
3. 测试标准
在测试计划或者测试方案中制订的测试标准。
4. 项目验收标准
在《软件开发合同》中制订的验收标准。
